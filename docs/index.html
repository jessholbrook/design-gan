A General Purpose Research Experiment · autoresearch × design

Auto improving websites and conversations with agents.

design-gan is a dual-agent loop: a generator writes a single-page website from a brief, a critic scores it on the System Usability Scale, and the feedback rolls forward into the next draft — until the composite score plateaus.

The idea

Inspired by Andrej Karpathy's autoresearch — a single agent autonomously evolving an LLM training script against a deterministic loss. This experiment takes the same autonomous-loop paradigm and applies it to artifacts that don't have a clean numeric metric: single-page websites and short conversations.

Two main differences. First, we split the work into a generator and a critic — hence the “gan” in the name — rather than running one agent that both modifies and evaluates. Second, the score is a Claude critic's verdict on a standard usability questionnaire, anchored by objective penalties (axe-core violations for websites; repetition and boilerplate in conversations), because there's no bits-per-byte to minimize.

A generator agent writes a standalone HTML document (or a conversation assistant's system prompt) from the brief. A headless Chromium renders the page — or a user-simulator agent runs the dialogue — and the rendered artifact goes to the critic. The critic fills out the 10-item System Usability Scale (or the parallel Conversation Usability Scale), writes feedback, and returns a short list of concrete next-step suggestions that feed the next generator call. The loop stops after a configurable window of iterations produce no meaningful composite-score gain, or at a hard iteration cap.

The composite score is the standard SUS score (0–100) minus a weighted axe-core penalty. The accessibility penalty is capped — but it keeps the critic honest relative to visual appeal - e.g. a solid design can't hide a colour-contrast failure or missing alt text.

A run, iteration by iteration

Brief: “A landing page for a weekend cycling tour in rural Vermont.”

Four iterations of the real loop against one brief — pages you could imagine coming upon in the real world. The composite score climbed from a competent-but-flawed baseline (52) to a polished peak at iteration 3 (76), then slipped back as the critic tightened its stance on colour-contrast failures — a textbook local-optimum trajectory rather than a monotonic ascent. Now the same loop, for conversations

design-gan isn't about websites — it's about treating any generated artifact as something you can evolve under critique. The second half of the system runs the identical loop over conversations: a user simulator plays a person with a goal, the assistant replies over up to five turns, and the critic scores the whole transcript on a parallel Conversation Usability Scale (CUS).

The composite score formula is unchanged: CUS (0–100) minus weighted objective penalties (repetition, boilerplate phrases, length bloat, unresolved goals), with the penalty capped at 30 so reasoning can't be buried by nitpicks.

The user simulator is another Claude — same model as the assistant — given a persona sketch and the goal. It decides when it's satisfied and ends the conversation naturally, so turn count is an output, not just a cap.

The only minor drag is the opening clarification turn added a round-trip before anything actionable landed, though the payoff — advice personalized to 'Gen Z political polarization' rather than a generic thesis — clearly justified it. Tone & register: The assistant executed nearly everything right: it asked exactly two clarifying questions, used the user's real thesis topic to make the bad-sentence template concrete, named the psychological mechanism without padding it, acknowledged the user's win in a single clause, and closed each turn with a crisp push. The user reached explicit resolution in three assistant turns with no hedging, no filler openers, and no repeated advice. Minor deductions: the two-question opener reads slightly script-like rather than conversational, and the closing push ('Go. Start the timer as soon as this window closes.') feels templated rather than earned. Trust & specificity: The assistant executed nearly perfectly: it gathered exactly the two data points it needed, named the real psychological block with specificity, and made the intervention concrete by using the user's actual thesis topic in the example sentence. Acknowledging the win in one clause before pivoting to the next tactic was crisp and non-patronizing. The only hairline crack is that the user had to ask 'now what?' — a single forward-pointer at the end of turn 3 (e.g., 'once you have it, we'll set a 25-min timer') could have anticipated that and kept momentum tighter.

Suggestions sent to the next generator

  • Consider offering a short provisional hypothesis *while* asking the two questions, so the user gets something actionable even if they don't reply — reduces the cost of the clarification round-trip.
  • The 'write one deliberately bad sentence' tactic could include a brief instruction on where to put it (top of the doc, new file, notes doc) to remove the micro-decision that can itself stall action.
  • When the user reported the sentence worked 'a little,' acknowledging the partial success rather than a full win ('Good — that's the hardest move') more honestly mirrors their experience and could deepen trust.
  • The closing push ('Go. Start the timer as soon as this window closes.') is strong but could name a specific timer tool or method (phone clock, Pomofocus, etc.) for users who would otherwise spend time choosing one.
  • Trust calibration: the analytical-vs-generative claim is stated as fact with no hedging; a brief attribution (even 'cognitive science of writing suggests…') would make it more credible without adding filler.
  • Weave the two clarifying questions into a single natural sentence rather than a numbered list — it reads more like a coach and less like a chatbot intake form.
  • Make the bad-sentence template even rougher ('I think social media does something to Gen Z politics, but I'm not sure what') — a more obviously terrible example lowers the bar further and is easier for the user to beat.
  • After the 25-minute timer suggestion, give the user one post-timer decision rule unprompted (e.g., 'If you filled more than half a page, switch to editing next session; if not, do another write round') — it prevents the next stall before it happens.
  • When crediting Anne Lamott, briefly anchor why it's trustworthy ('her actual term is shitty first drafts — it's from a widely-used craft book') so the user knows it's a real, lookable-up concept rather than a coaching platitude.
  • Consider proactively naming what 'after the 25 minutes' looks like in one clause — users often stall again at the transition, and pre-empting it keeps momentum without adding length.

Design notes

  • The critic sees the rendered page, not the code. Code-only critique is cheap but correlates poorly with real usability.
  • Subjective + objective. SUS alone is gameable; axe-core anchors the score to measurable a11y signals.
  • Convergence, operationalised. “No further improvements” = patience iterations in a row without a composite gain of at least tolerance points.
  • Structured output. The critic uses Pydantic-validated JSON so a malformed response triggers a retry with a stricter nudge.
  • Prompt caching. System prompts are cached as ephemeral, so iteration N+1 pays near-zero for the static instructions.
